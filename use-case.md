# Problem Statement: Daily AI Trends Publishing Site

## Background

A scheduled task ("tech-ai-trends") already runs each evening and produces a curated digest of tech/AI trends from X (Twitter), gathered via web search since there's no direct X/Twitter API access. Each run writes:

- A dated markdown file (`YYYY-MM-DD-x-tech-ai-trends.md`) with 5-8 curated trend items, each including a title, a short "why it's trending" explanation, and a source link.
- An `INDEX.md` with one summary row per day, linking to that day's file.

Currently these files are saved to a Google Drive folder . This works as an archive, but it's not a readable "website" — there's no public/shareable URL, no rendered HTML, and browsing a folder of markdown files (or a growing INDEX.md table) doesn't scale well as a daily habit.

## Problem

We want a lightweight, low-maintenance website where each day's AI/tech trends digest is automatically published and viewable as a clean web page — something we can just open and read, instead of digging through files in a Drive folder.

Key pain points to solve:

1. **No publishing target today.** The digest is generated correctly but only lives as markdown files in cloud storage — not rendered anywhere as a browsable site.
2. **Long-term scale.** Over a year this becomes ~365 dated files. Flat-folder storage and a single hand-maintained index table won't stay easy to browse — we need a structure that scales without becoming unwieldy (e.g., organized by year/month, with auto-generated navigation/pagination rather than manual index upkeep).
3. **Automation.** The digest is generated by an unattended scheduled task (runs daily, no human in the loop). Whatever publishing mechanism we pick must be something that task can push to automatically — no manual copy-paste step each day.

## Proposed Direction (to validate/build)

- **Storage + hosting:** A GitHub repository (private), e.g. `ai-updates, using GitHub Pages for hosting. GitHub comfortably handles this scale — a year of small daily markdown files is only a few MB total, far below any repo size limits.
- **Structure:** Use Jekyll's built-in blog/`_posts` structure (native to GitHub Pages, no extra build tooling required) — one file per day in `_posts/`, with the date in the filename. Jekyll auto-generates the homepage, pagination, and an archive by month/year, so we don't have to hand-maintain an index as the file count grows.
- **Automation:** Each scheduled run of the digest task should commit the new day's file to the GitHub repo (via the GitHub connector/API) so the published site updates itself with no manual step.

## Goal

A working pipeline where: scheduled task runs daily → generates the curated tech/AI trends digest → publishes it as a new dated post on a GitHub Pages site → the site is browsable at a stable URL with automatic navigation across days/months, with no manual intervention required after initial setup.
